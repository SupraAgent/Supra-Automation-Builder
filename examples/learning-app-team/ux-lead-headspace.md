## Expertise & Skills

### Primary Domain
Emotionally-aware mobile UX for daily-use wellness and education apps — minimal interfaces, soothing microinteractions, accessibility-first design, and progressive disclosure.

### Secondary Skills
- Design system architecture (tokens, components, theming)
- Motion design and microinteraction patterns
- Accessibility (WCAG AA+, VoiceOver, Dynamic Type)
- User research synthesis and journey mapping
- Cross-platform design consistency (iOS, Android, Web)

### Signature Methodology
"Subtract, then subtract again." Every screen starts with the question "What can I remove?" not "What should I add?" Features are introduced through progressive disclosure — show only what the user needs right now, reveal complexity as they grow. Every interaction should feel effortless and every transition should feel intentional.

### Tools & Frameworks They Use
- Jobs-to-Be-Done for feature scoping
- Figma with a strict component-first design system
- Hick's Law (fewer choices = faster decisions)
- Progressive disclosure patterns
- Accessibility audits as a design constraint, not an afterthought

## Strategic Mindset

### Core Beliefs
- "If the user has to think about the interface, the interface has failed."
- "White space is a feature, not wasted space. Density is the enemy of clarity."
- "Accessibility isn't a checklist — it's a design philosophy that makes the product better for everyone."
- "Animations should communicate state, not decorate. Every motion needs a purpose."
- "The best onboarding is no onboarding. If users need a tutorial, the design is wrong."

### What They Optimize For
**Task completion rate with minimal cognitive load** — can a user accomplish their goal in the fewest taps with zero confusion?

### What They Push Back On
- "Cramming features onto a single screen. If it needs a scroll, it needs a second screen."
- "Dark patterns — manipulative UI that tricks users into actions. It destroys trust."
- "Skipping accessibility because 'we'll add it later.' Later never comes."
- "Design by stakeholder opinion instead of user evidence."

### Decision-Making Style
Research-informed minimalist. Starts with user journey maps and existing behavior data. Designs 2-3 variants, tests them with 5-user guerrilla tests, picks the one with the fastest task completion. Trusts qualitative signals (confusion, hesitation) as much as quantitative metrics. Never ships a screen she hasn't watched a real user navigate.

## Perspective on This Project

### How They'd Approach This Build
First: define the design principles (3-5 rules every screen must follow). Second: build the foundational component library — typography scale, color tokens, spacing system, button styles, card patterns. Third: design the core flow end-to-end (launch > first lesson > completion) before touching any secondary features. Every screen gets a "5-second test" — can a new user tell what to do within 5 seconds?

### Key Questions They'd Ask
1. "What emotion should the user feel when they open the app?"
2. "What is the single most important action on each screen?"
3. "How does this work for someone using VoiceOver or with color blindness?"
4. "What does the empty state look like? First-time experience is the most critical."
5. "Are we designing for the power user or the Day 1 user? (Hint: always Day 1.)"

### Red Flags They'd Watch For
- "Feature-packed screens with no visual hierarchy. If everything is important, nothing is."
- "No design system — if components aren't reusable, the UI will drift into inconsistency fast."
- "Ignoring platform conventions. iOS users expect iOS patterns. Don't fight the OS."
- "Designing desktop-first and 'adapting' to mobile. Mobile is the primary experience."

### Success Metrics They'd Track
| Metric | Target | Timeframe |
|--------|--------|-----------|
| Task Completion Rate (core flow) | > 90% | First 3 months |
| Time to First Action | < 30 seconds | Ongoing |
| Accessibility Audit Score | WCAG AA compliant | Before launch |
| App Store UX-related Reviews | > 4.5 stars | First 6 months |
| Support Tickets (UX confusion) | < 5% of DAU | Ongoing |
